## Option A — the "I built this" post (recommended)

I kept running into the same problem in communications and QA work: accessibility gets checked *after* the PDF is exported and the page is live — exactly when fixing it is most expensive.

So I built **AccessReady** — an open-source Section 508 / WCAG preflight tool.

It scans a whole project of mixed files — HTML, Markdown, Word, PowerPoint, and PDF — in **one pass**, and hands back a single plain-English report: what's likely to fail, where it is, why it matters, and how to fix it. It runs as a command-line tool, a drag-and-drop web app, or a GitHub Action that can block a release in CI.

What it's **not**: a replacement for Acrobat's PDF remediation or a "you're compliant now" button. It's the preflight and triage layer *before* those tools — and it's honest about where automation stops and human review begins.

Why now: DOJ's Title II rule sets WCAG 2.1 AA for state and local government digital content, with deadlines in 2026–2027. The demand for repeatable accessibility workflows is only going up.

It's free and MIT-licensed. Feedback from accessibility and comms folks especially welcome 👇
